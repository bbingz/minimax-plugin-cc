## Task A.1 — Fix Critical: `$&` / `$$` replace corruption (Gemini finding)

**Why:** `String.prototype.replace(stringPattern, stringReplacement)` interprets `$&`, `$$`, `` $` ``, `$'`, `$1`-`$9` in the replacement string EVEN WHEN the pattern is a string. If a user's diff contains `arr.replace(re, '$& foo')` literal, our `staged.replace(SENTINEL, context)` will replace `$&` with the matched sentinel string, silently corrupting the prompt sent to Mini-Agent.

**Affected sites in `plugins/minimax/scripts/lib/minimax.mjs`:**
- `buildReviewPrompt`: 4 user-derived `.replace()` calls (CONTEXT_SENTINEL, SCHEMA_JSON, FOCUS, RETRY_HINT)
- `buildAdversarialPrompt`: 5 user-derived `.replace()` calls (STANCE_INSTRUCTION trusted internal, but CONTEXT_SENTINEL, SCHEMA_JSON, FOCUS, RETRY_HINT all touch user-derived data)

**Fix strategy:** Replace `String.prototype.replace(string, string)` with `String.prototype.replaceAll(string, () => value)` style — the function-callback variant disables `$X` interpretation. We use an arrow function returning the value, which is byte-safe.

Why callback over `split().join()`: callback preserves first-match semantics (replaces only the first occurrence) which we WANT for `{{X}}` placeholders to avoid double-replacement bugs.

Wait — `replaceAll` replaces ALL occurrences. We want `replace` (first only) with callback semantics. The callback form `replace(pattern, () => value)` is correct: matches first, calls callback with no `$` interpretation.

## Task A.5 — `extractReviewJson` depth-goes-negative permanently fails (Gemini High)

**Why:** `extractReviewJson`'s brace-balanced scanner decrements `depth` on every `}`. Subsequent `{` increments depth back to 0, but the `start` pointer for the JSON object is set when going from depth=0 to depth=1 via INCREMENT — if it's set when depth WAS 0 going to 1, fine. But if depth was -1 going to 0 (unbalanced state), then 0 to 1, the start may be set incorrectly OR the closing detection triggers wrong.

**Need to verify:** Read the actual `extractReviewJson` implementation, find the depth logic, identify the bug, then patch.
