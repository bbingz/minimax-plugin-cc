## 1. Scope (all three ship together in v0.1.3)

1. **Timing telemetry**: `lib/timing.mjs` + `timings.ndjson` history + `/minimax:timing` command. Field names match `gemini-plugin-cc/plugins/gemini/scripts/lib/timing.mjs::TimingAccumulator.build()` output; fields we cannot physically fill (no stream events from Mini-Agent) are explicit `null`, EXCEPT array-typed fields which stay empty-array (see §4 same-name-different-meaning callout). 2. **SessionStart cleanup**: extend `session-lifecycle-hook.mjs` to prune `~/.claude/plugins/minimax/jobs/mj-*/` by mtime > 3 days, AND add SessionEnd per-session-id cleanup mirroring Gemini's pattern adapted to our per-job-directory layout.
3. **Upstream Mini-Agent issue**: file a single issue at `MiniMax-AI/Mini-Agent` with three tiered asks (P0 served_model / P1 usage / P2 per-line timestamps). File **immediately** on v0.1.3 kickoff — not blocked on release.

The three items are mutually independent at the code level but bundled into one release because they share one cross-plugin re-alignment cycle. See §8 for file changes and §9 for test/hard-gate strategy.

## 2. Why these three, why now

- **Timing**: Gemini's P0 finding was "timing 完全缺席". Verified by grep — `timing|appendTiming|TimingAccumulator|timings\.ndjson` has 0 hits anywhere in `plugins/minimax/scripts/`. This is not a stub; it is absent. The user-facing question "is MiniMax actually fast?" is unanswerable without it.
- **SessionStart cleanup**: Gemini's P2 finding flagged `session-lifecycle-hook.mjs` as suspiciously small (30 lines vs their 114). Verified — we do only env injection and session-id file delete. Our `~/.claude/plugins/minimax/jobs/` grows unbounded; Phase 4 never wired cleanup. Left alone, a heavy user hits inode exhaustion eventually.
- **Upstream issue**: P1 (primary-model attestation) is blocked until Mini-Agent exposes a `served_model` field in the log RESPONSE block. Our own v0.1.3 timing cannot fill `usage` / `tokensPerSec` / silent-fallback-detection fields without this upstream change. Filing the issue now maximizes the window for upstream feedback before v0.2.

Gemini's re-alignment specifically compares along these three axes; matching field names and picking 3 days as the mtime threshold closes two of them without a second round-trip. See memory `project-v013-realignment.md`.

## 3. Architectural decisions

### D1. ndjson schema = Gemini's `timing.build()` keys, honest `null` for unfillable scalars, empty-array for unfillable arrays

**Decision**: our `TimingAccumulator.build()` returns a superset of Gemini's 18-field object (`gemini/plugins/gemini/scripts/lib/timing.mjs:147-168`). Plus one discriminator `invariantKind: "3term"` we add. Fields we can compute from post-hoc log parse have real values; unfillable scalar fields are explicit `null`; unfillable array-typed fields are `[]` (matching Gemini's `usage: this._usage || []` pattern at `timing.mjs:137`). `coldStartPhases` stays `null` — Gemini also returns null when not set.

**Rationale**:
- Re-alignment requires schema compatibility → field-name drift is the single biggest risk → name-match verbatim.
- Array-vs-null matters: Gemini's downstream uses `Array.isArray(usage) && usage.length > 1`. Null array breaks `.map()` / `.length` in any downstream code that forgets to guard; empty-array stays safe.
- Three fields (`firstEventMs`, `streamMs`, `retryMs`) share Gemini's names but carry different semantic meaning in our backend — documented in the §4 compat callout. Do NOT attempt cross-plugin numerical comparison on these three without adjustment.

**Trade-off**: our output is visually noisier (many `null` fields). Accepted — readability is fixed at the render layer (`renderHistoryTable` prints `—` for null), not the storage layer.

### D2. TimingAccumulator has all Gemini methods; the stream-path ones are empty no-ops

**Decision**: `onFirstToken`, `onLastToken`, `onToolUseStart`, `onToolResult`, `onRetryStart`, `onRetryEnd`, `onStartupStats`, `onResult`, `dispatchTimingEvent` all exist as empty methods with pinned contract:
- Signature: `(event?) => void`
- Body: `return;` (no side effects, no `this` chaining, no logging, no throw)
- `@returns {void}` JSDoc annotation
- Test gate: callable without throw; future impls MUST preserve void return even when adding real behavior

**Rationale**: if Mini-Agent upstream eventually emits stream events (upstream P1/P2 asks, see §10), we can wire them in by implementing existing methods rather than refactoring the class. Pinning the return contract now means future dispatcher code stays forward-compatible.

**Trade-off**: adds ~40 lines of dead code today. Accepted — significantly less churn than adding the methods later.

### D3. Dual cleanup in session-lifecycle-hook

**Status enum reference**: `plugins/minimax/scripts/lib/job-control.mjs::VALID_STATUSES = {queued, starting, running, done, failed, canceled}`. This spec uses THOSE values, NOT the English names (no `completed` / `cancelled`).

**Decision**:
- `SessionEnd`: scan `~/.claude/plugins/minimax/jobs/mj-*/`; for each dir read `meta.json`; delete directories where `meta.sessionId === currentSessionId` AND `meta.status ∈ {done, failed, canceled}`. Preserve `running`, `queued`, `starting` (detached-worker semantic from Phase 4 — a `_worker` may still be mid-execution).
- `SessionStart` (stale sweep):
  - Scan dir; per entry decide by `(meta.status, dir-mtime)`:
    - **terminal status** (`done|failed|canceled`) AND mtime > 3 days → `rmSync` (safe: already terminal, no process concern).
    - **non-terminal status** (`running|queued|starting`) AND mtime > 3 days → do NOT trust mtime alone. Require TWO conditions: (a) `process.kill(meta.pid, 0)` throws ESRCH (process gone), AND (b) the dir's own mtime > 3 days. Only when both hold, `rmSync`. Live process → skip. Missing `meta.pid` → skip (cannot verify).
    - **missing `meta.json` (ENOENT)** AND dir-mtime > 3 days → `rmSync` (no meta, stale dir → orphan).
    - **corrupt `meta.json` (JSON parse throws)** AND dir-mtime > 3 days → `rmSync` (per v1 C10; stale corrupt meta is almost certainly an abandoned write from a dead worker).
    - **corrupt `meta.json` AND dir-mtime ≤ 3 days** → **skip** (v2.1: may be a mid-write from an active concurrent `createJob`; do not touch). Filesystem updates parent-dir mtime on child-file create/write, so fresh dir-mtime with corrupt meta = active writer; stale dir-mtime with corrupt meta = abandoned.
    - Any mtime ≤ 3 days → preserve regardless of status (the 3-day gate IS the concurrent-write protection).

**Rationale**:
- SessionEnd handles happy path (session exits cleanly → its terminal jobs gone).
- SessionStart handles sad path (prior session crashed before SessionEnd → jobs orphaned forever without mtime sweep).
- 3-day threshold matches Gemini's alignment suggestion — zero re-alignment friction.
- **pid-reuse caveat corrected from v1**: v1 argued "3 days old makes pid reuse vanishingly unlikely" — that's backwards (older = MORE chance pid was reused). Corrected: the mtime sweep only evicts non-terminal jobs when BOTH the process is gone AND the dir is old, so a reused pid belonging to an unrelated process still protects the dir (kill(pid,0) succeeds → we skip). Phase 4's pid-reuse caveat in `cancelJob` is preserved.
- Terminal-state jobs are safe to delete without any pid check (meta recorded exit before this hook runs).
- Corrupt meta.json path: `readJob()` throws on JSON parse failure; the sweep wraps it in try/catch and treats throw-result as "missing meta" for the decision table above.

**Trade-off**: cleanup runs on every SessionStart, adding a few ms to session boot. Accepted — scan is `listJobs()` + `statSync` per dir, bounded by number of lifetime-recent jobs (typically <100).

### D4. Stale threshold default 3 days, env-overridable

**Decision**: `const STALE_JOB_THRESHOLD_MS = Number(process.env.MINIMAX_STALE_JOB_THRESHOLD_MS) || (3 * 24 * 60 * 60 * 1000)` in `session-lifecycle-hook.mjs`.

**Rationale (corrected from v1)**: v1 argued YAGNI, but v1 rationale assumed users could "edit the constant" — they can't, the plugin file gets overwritten on `/plugin update`. Env override is 3 lines of code, near-zero maintenance surface, and bad values (`NaN` / non-numeric) fall back to default via `||`.

**Trade-off**: one env var to document. Accepted — cheaper than forcing users to unofficial workarounds. `env.MINIMAX_STALE_JOB_THRESHOLD_MS` documented in `/minimax:timing` command help and in `README.md` env-vars section.

### D5. `/minimax:timing` offers three views: default history table, `--aggregate` percentiles, `--json` raw

**Decision**: three output modes, no more.

**Rationale**:
- History table answers "what did my recent runs look like?" — the most common question.
- Aggregate answers "what's the p95 latency for `/minimax:ask`?" — the diagnostic/tuning question.
- `--json` lets pipelines / other tools consume raw records.
- Watch mode, SVG export, cross-plugin compare all defer to v0.2 if demand materializes.

### D6. NDJSON file at `${CLAUDE_PLUGIN_DATA}/timings.ndjson`; jobs dir stays at legacy path

**Decision**: new `timings.ndjson` goes to the Claude-Code-standard plugin data path (env `CLAUDE_PLUGIN_DATA` with `~/.claude/plugins/data/minimax-minimax-plugin/` fallback). The existing `~/.claude/plugins/minimax/jobs/` stays where Phase 4 put it.

**Rationale**: Gemini uses the standard path for their ndjson → matching it is cheap compat. Migrating the jobs dir is v0.2 scope (risky, touches every `/minimax:*` command that reads a job), out of v0.1.3 scope.

**Trade-off**: ndjson and jobs live under different roots. Documented in §8 and in `lib/state.mjs` comment.

### D7. Adversarial review emits two ndjson records (one per stance), not one merged

**Decision**: `callMiniAgentAdversarial` runs red then blue sequentially; each spawn gets its own `TimingAccumulator`; each writes its own record with `kind: "adversarial-red"` / `kind: "adversarial-blue"`.

**Rationale**: the two spawns have different prompts, different response times, different failure modes. A merged record would hide that blue is systematically faster than red (or vice versa). Downstream aggregates can filter by `kind` to see per-stance distributions.

**Aggregate composition rule (added v2)**: `--aggregate` MUST specify a `--kind` other than `all`. Passing `--kind all` or omitting it when multiple kinds exist is an error — exit 2 with `usage: --aggregate requires --kind <single-kind>`. Reason: mixing `ask` (unimodal) and `adversarial-red + adversarial-blue` (bimodal-by-design, doubled count) produces statistically meaningless aggregates. Users wanting "all adversarial" data run the command twice (once per stance) or read the raw `--json`.

**Trade-off**: two rows per adversarial invocation; `--aggregate --kind all` is blocked. Accepted — users thinking "one adversarial call should be one row" get the answer "it's two spawns, each costed separately"; users wanting cross-kind rollups must pick one kind at a time.

## 4. ndjson record shape

```json
{
  "_v": 1,
  "jobId": "mj-<uuid>",
  "kind": "ask" | "review" | "adversarial-red" | "adversarial-blue" | "rescue",
  "ts": "2026-04-23T10:00:00.000Z",
  "timing": {
    "spawnedAt":         "2026-04-23T09:59:55.123Z",
    "firstEventMs":      156,         // spawn -> first stdout line. SEE §4 COMPAT CALLOUT: this measures CLI boot (Python+click+skill-metadata), NOT model TTFT. True TTFT blocked on upstream P0.
    "ttftMs":            null,        // no stream events; distinct from firstEventMs by design
    "streamMs":          12800,       // first stdout line -> last stdout line. SEE §4 COMPAT CALLOUT: not comparable to Gemini's firstToken->lastToken.
    "toolMs":            null,        // no tool events
    "retryMs":           null,        // SEE §4 COMPAT CALLOUT: Mini-Agent auto-retries 1+2+4s on 401 (probe P0.2/P0.5) — silent retry padding IS absorbed into streamMs.
    "tailMs":            76,          // last stdout line -> process close. Log-parse cost is downstream of this.
    "totalMs":           13032,       // close - spawnedAt == firstEventMs + streamMs + tailMs (clean exit)
    "promptBytes":       3421,
    "responseBytes":     8912,
    "exitCode":          0,
    "terminationReason": "exit" | "timeout" | "signal" | "error",
    "timedOut":          false,
    "signal":            null,
    "requestedModel":    "MiniMax-M2.7-highspeed",
    "usage":             [],          // array always; empty until upstream P0 ships served_model + token counts
    "tokensPerSec":      null,        // derivable from usage once populated
    "coldStartPhases":   null,        // matches Gemini: null when startup_stats event absent
    "invariantOk":       true,        // populable: firstEventMs + streamMs + tailMs === totalMs on clean exit
    "invariantKind":     "3term"      // v0.1.3-specific discriminator; Gemini-equivalent would be "6term"
  }
}
```

### §4 compat callout — three same-name-different-meaning fields

These three field names are shared with `gemini-plugin-cc`'s `timing.build()` output but carry **different semantic meaning** in our backend. Cross-plugin numerical comparison on these fields is INVALID without adjustment:

| Field | Gemini semantic | minimax semantic | Why they differ |
|---|---|---|---|
| `firstEventMs` | spawn → first NDJSON event (≈ model cold start surface) | spawn → first CLI stdout line (Python+click+skill-metadata boot) | Mini-Agent emits ~26 banner lines before any LLM output; probe P0.1 |
| `streamMs` | firstToken → lastToken (excl. tool/retry windows) | first stdout line → last stdout line (INCLUDES silent retry padding) | no per-token events; Mini-Agent retries 1+2+4s on 401 invisibly; probe P0.2/P0.5 |
| `retryMs` | sum of retry windows observed in stream | null — retries are opaque; padding absorbed into `streamMs` above | Mini-Agent does not emit retry boundary events |

**Implication**: v0.1.3 aggregates answer "is minimax run A faster than minimax run B" correctly (same semantic within plugin), but do NOT answer "is minimax faster than gemini" on these three fields. Cross-plugin comparison MUST use `totalMs` (true wall time, semantic identical) or wait for upstream P0 to enable stream-level parity.

### Other null fields (uncontroversial)

- `ttftMs`, `toolMs`, `coldStartPhases` → no stream events → physically unavailable (matches Gemini's null when event absent).
- `tokensPerSec` → derivable once `usage` populates after upstream P0.
- `invariantOk` / `invariantKind` → **populable**: `invariantOk = (firstEventMs + streamMs + tailMs === totalMs)` on clean exit, `null` otherwise. `invariantKind: "3term"` tells downstream tools which sum was checked. Gemini's invariant is `"6term"` (includes `ttftMs + toolMs + retryMs`); a cross-plugin tool comparing records must branch on `invariantKind` rather than assume shape.

### Derived render column `fb` (fallback flag)

The history table (§7) column `fb` is render-time-derived from `usage`:
- `'y'` if `usage.length > 1` (silent fallback detected: served model differed from requested)
- `'n'` if `usage.length === 1`
- `'—'` if `usage.length === 0` (v0.1.3 state; pending upstream P0)

Rendering `'—'` when unknown is intentional — a hardcoded `'n'` would be misleading.

### Integration — split between `callMiniAgent` (timing collection) and callers (timing write)

**v2 correction**: v1 put `appendTimingHistory` inside `callMiniAgent`. That was wrong on two counts: (a) `appendTimingHistory` does synchronous I/O (O_EXCL lock spin + appendFileSync + potential half-trim) which v1 mis-described as "fire-and-forget"; (b) mirrors Gemini's pattern where their `gemini-companion.mjs:236` / `:550` calls `appendTimingHistory` AFTER the streaming call returns, not inside it. v2 splits responsibilities: `callMiniAgent` produces a `timing` snapshot; caller decides when and whether to persist.

#### `callMiniAgent` signature change

```js
// OLD signature (v0.1.2):
export async function callMiniAgent({ prompt, timeout, onStdoutLine, ...rest }) { ... }

// NEW signature (v0.1.3):
export async function callMiniAgent({ prompt, timeout, onStdoutLine, jobId, kind, ...rest }) {
  const timing = new TimingAccumulator({ spawnedAt: Date.now(), prompt });
  timing.setRequestedModel(readMiniAgentConfig().model);  // NOT readRequestedModel — uses existing helper

  let firstEventSeen = false;
  const wrappedOnStdoutLine = (line) => {
    if (!firstEventSeen) { timing.onFirstEvent(); firstEventSeen = true; }
    timing.onStdoutLine();  // updates lastEvent for streamMs/tailMs calc
    onStdoutLine?.(line);
  };

  const { exitCode, timedOut, signal } = await spawnWithHardTimeout({
    /* ... */,
    onStdoutLine: wrappedOnStdoutLine,
  });
  timing.onClose(Date.now(), { exitCode, timedOut, signal });

  const parsed = await parseLogFileAndClassify(/* ... */);
  timing.recordResponseBytes(parsed.responseBytes || 0);

  // v2: return timing in the result envelope; caller decides persistence.
  // `jobId` / `kind` default to null so legacy callers that haven't been updated
  // (or tests mocking the function) still get a well-typed envelope.
  return { ...parsed, timing: timing.build(), jobId: jobId ?? null, kind: kind ?? null };
}
```

**Documented limitation (Codex v2.1 finding)**: if `callMiniAgent` throws (rather than returns a structured error envelope) before the final `return`, no timing record is written for that call. This is acceptable for v0.1.3 — throws from `callMiniAgent` are rare (spawn failures classify through the return path, not throws) and typically surface as fatal caller errors the user will notice. A `try/finally` emitting partial telemetry would add complexity for edge-case value; deferred to v0.2.

#### Caller-side persistence (in each call-site wrapper)

```js
// e.g. in runAsk / runReview / runRescue / callMiniAgentAdversarial red-loop / blue-loop:
const result = await callMiniAgent({ /* ... */, jobId, kind });
// Only now write telemetry, after primary classification complete.
// try/catch because appendTimingHistory IS synchronous I/O; never let it block primary result.
try {
  appendTimingHistory({
    _v: 1,
    jobId: result.jobId,
    kind: result.kind,
    ts: new Date().toISOString(),
    timing: result.timing,
  });
} catch (e) {
  // Log to stderr at debug level; do not alter primary result.
}
return result;
```

#### Call sites requiring `{jobId, kind}` propagation

v1 §8 counted this as "+20 lines in callMiniAgent". v2 corrects: the signature change touches **five distinct call sites** in addition to `callMiniAgent` itself:

| Call site | Location | `kind` | `jobId` source |
|---|---|---|---|
| `/minimax:ask` foreground | `minimax-companion.mjs::runAsk` | `"ask"` | synthetic (`mj-` + randomUUID, telemetry-only, NOT in job-control) |
| `/minimax:review` + retry | `lib/minimax.mjs::_callReviewLike` (called from `callMiniAgentReview`) | `"review"` | synthetic |
| `/minimax:adversarial-review` red | `lib/minimax.mjs::callMiniAgentAdversarial` (red branch) | `"adversarial-red"` | synthetic, distinct from blue |
| `/minimax:adversarial-review` blue | `lib/minimax.mjs::callMiniAgentAdversarial` (blue branch) | `"adversarial-blue"` | synthetic, distinct from red |
| `/minimax:rescue` (fg + bg _worker) | `minimax-companion.mjs::runRescue` + `_worker` subcommand | `"rescue"` | **real** job id from `createJob()` |

**Synthetic jobId rule** (D6-adjacent, new in v2): for kinds other than `"rescue"`, `callMiniAgent` (or its caller) generates `"mj-" + randomUUID()` purely for telemetry joining. These ids do NOT appear in `job-control` / `/minimax:status`. Users querying `/minimax:status mj-abc123` for a telemetry id get `no such job`. This separation keeps v0.1.3 scope small — migrating every call to go through `createJob()` is v0.2.

**Retry case**: review/adversarial retries share the same `jobId` across first-shot + retry spawn; two records emitted under the same jobId. Downstream `--json --jobId mj-X` shows both.

## 6. appendTimingHistory (`lib/state.mjs`)

Port of Gemini's `appendTimingHistory` at `gemini-plugin-cc/plugins/gemini/scripts/lib/state.mjs:274-328`:

- Path: `${CLAUDE_PLUGIN_DATA}/timings.ndjson` or fallback `~/.claude/plugins/data/minimax-minimax-plugin/timings.ndjson` (matches Gemini `state.mjs:234-238`).
- Lock: O_EXCL `timings.ndjson.lock` sibling, 10s acquire timeout, 30s stale-lock reclaim (Gemini `state.mjs:246-270`).
- Crash recovery: before append, stat the existing file; if size > 0 and the LAST byte is not `\n` (detected via positional read), prepend `\n` to the write payload. Port of Gemini's stat-and-peek at `state.mjs:287-301` — our port should not introduce divergence.
- Retention: if post-append file size > `TIMING_MAX_BYTES = 10 * 1024 * 1024`, split on `\n`, keep only lines that `JSON.parse` successfully, retain **newer half by file-order position** (= append order under the lock; NOT sorted by `ts`), atomic `rename(tmp, file)` (Gemini `state.mjs:305-325`).
- Lock acquire timeout → log to stderr (`[timing] lock acquire timeout; dropping record mj-...`) and return `false`; never throw FROM the lock path — telemetry must not affect primary result. Caller observes the boolean return to detect drops.

**Contract change from v1**: v1 said "log and drop" generically. v2 adds: `appendTimingHistory` returns `boolean` (`true` = record appended, `false` = dropped due to lock timeout or trim error). Callers are expected to ignore the return for hot-path safety but MAY surface drops in diagnostic logs. Concurrent append test asserts **both writers succeed** under normal contention (not "one may be dropped"), matching Gemini's `tests/timing-storage.test.mjs` expectation.

**v2.1 clarification (Claude code-reviewer M1)**: `appendTimingHistory` CAN still throw from **pre-lock paths** — `mkdirSync(PLUGIN_DATA, {recursive:true})` on permission errors (EACCES), `statSync` on existing file (EACCES), and `renameSync(tmp, file)` during trim (EACCES/EXDEV cross-device). Post-lock paths go through the boolean-return contract; pre-lock and rename-during-trim are the throw paths. Callers' try/catch (§5) is therefore non-dead — it catches filesystem permission errors that can't be usefully recovered here.

## 7. `/minimax:timing` command

### Signature

```
/minimax:timing [--kind <ask|review|adversarial-red|adversarial-blue|rescue|all>]
                [--last <N>]             # default 20
                [--since <ISO-timestamp>]

**Rule (D7)**: `--aggregate` requires `--kind <single-kind>`. Passing `--aggregate` with no `--kind`, or `--aggregate --kind all`, exits 2 with:
```
/minimax:timing: --aggregate requires --kind <ask|review|adversarial-red|adversarial-blue|rescue>
  (adversarial emits two records per invocation; mixing kinds produces meaningless aggregates).

## 9. Test strategy + T14 hard gate

### Unit tests

**`timing.test.mjs`** (new, ~10 tests)
- `TimingAccumulator` spawn → firstEvent → multiple onStdoutLine → close yields correct `build()` values
- `firstEventMs`, `streamMs`, `tailMs` all `null` when spawn → close with no stdout
- `streamMs` = 0 when only one stdout line seen (first == last)
- `tailMs` clamped ≥ 0 when clock returns to earlier time mid-call
- `invariantOk` true when `firstEventMs + streamMs + tailMs === totalMs` on clean exit
- `invariantOk` null when terminationReason !== "exit"
- `onClose` covers exit / timeout / signal terminationReason values
- `recordResponseBytes` accumulates across calls
- Empty no-op methods (`onFirstToken` etc.) are callable and do not throw
- `computeAggregateStats` n<20 → p95 null; n<100 → p99 null
- `computeAggregateStats` fallback rate = 0 when all records have `usage: null`
- `renderHistoryTable` prints `—` for null fields
- `filterHistory` kind/since/last filters combine correctly

**`state.test.mjs`** (extend, +7 tests)
- `appendTimingHistory` basic append → each line JSON-parseable, returns `true`
- 10MB cap triggers half-trim → file size < cap → retained lines are newer half (by file-order position, not `ts`)
- Crash recovery: file without trailing `\n` → next append prepends one (verified by reading resulting file)
- **Concurrent append, both succeed (new v2, strengthened per Codex H1)**: two appenders contend on the lock; BOTH writers complete successfully; both records present in ndjson; no drop.
- Pathological: appender holds lock > 10s (simulated via fake fs wait) → second returns `false`, stderr logs drop, first's record intact.
- Empty 0-byte file on disk → next append succeeds without prepending `\n` (empty-file branch, per Qwen finding).
- Lock path resolves to `CLAUDE_PLUGIN_DATA` when env is set, fallback otherwise.

**`session-lifecycle-hook.test.mjs`** (new, ~9 tests)
- SessionStart: mtime > 3d + status ∈ `{done, failed, canceled}` → rmSync invoked (pid check skipped for terminal)
- SessionStart: mtime > 3d + status ∈ `{running, queued, starting}` + `kill(pid,0)` throws ESRCH → rmSync invoked
- SessionStart: mtime > 3d + status running + kill(pid,0) succeeds (alive) → skip
- SessionStart: mtime > 3d + meta.json missing (ENOENT) → rmSync invoked (orphan branch, new v2)
- SessionStart: mtime > 3d + meta.json corrupt JSON → rmSync invoked (stale abandoned; new v2)
- SessionStart: mtime ≤ 3d + meta.json corrupt JSON → **skip** (mid-write race protection; new v2.1)
- SessionStart: mtime < 3d → preserved regardless of status
- SessionStart: `process.env.MINIMAX_STALE_JOB_THRESHOLD_MS = 60000` → threshold switches to 1 minute (D4 env override, new v2). **Test uses `t.before`/`t.after` to snapshot+restore `process.env` value so concurrent tests don't leak the override (Codex v2.1 M finding).**
- SessionEnd: session-matched terminal job → rmSync; session-matched running/queued/starting job → preserved
- rmSync throws EACCES → hook exits 0, does not propagate

**Title**: `Make mini-agent log more consumable for downstream telemetry (3 optional additions)`

**Body outline**:

```markdown
## Context
minimax-plugin-cc (Claude Code plugin wrapping mini-agent) emits per-spawn timing
telemetry mirroring `gemini-plugin-cc`'s `stats.models` pattern. Three additions
to mini-agent's log output would let us attest served-model, compute throughput,
and give sub-second TTFT resolution. All three are optional; any one of them is
independently useful. Feel free to split into separate issues if it suits your triage.

## Asks, ordered by implementation cost (cheapest first)

### upstream P0 — per-line timestamps in log block headers
Today `~/.mini-agent/log/agent_run_*.log` files have a second-precision UTC
stamp in the filename but individual lines inside the structured blocks carry
no timestamps. A minimal addition would be:

  [1] REQUEST  @ 2026-04-23T10:00:00.123Z
  ─────
  ...
  [2] RESPONSE @ 2026-04-23T10:00:13.456Z
  ─────
  ...

This is a pure logger-side change in `agent.py::log_request` / `log_response`.
Zero billing exposure. Zero API contract change. Unblocks sub-second TTFT
approximation for any downstream telemetry consumer.

Optionally (even smaller ask): a `--debug-stream-events` flag that dumps the
upstream Anthropic-compatible SSE lines already consumed internally to
stderr/sidecar. This would let downstream fill ALL the ttft/tool/retry fields
we currently null.

### upstream P1 — `served_model` field in RESPONSE block
The upstream `/anthropic` response likely already carries `x-served-model` or
an equivalent header; mini-agent just doesn't parse and log it. Add one field:

  { "content": "...", "finish_reason": "end_turn", "served_model": "MiniMax-M2.7-highspeed" }

Motivates: silent-fallback detection ("user requested model X, MiniMax actually
served model Y"). Low billing sensitivity — model name is already in user's
request path.

### upstream P2 — token usage counts in RESPONSE block (optional; opt-in)
  { ..., "usage": { "input_tokens": 1234, "output_tokens": 5678, "thoughts_tokens": 0 } }

We recognize that `usage` counts are billing-sensitive and may not be appropriate
for default log output. If that's a concern, a config-gated opt-in would still
unblock downstream cost telemetry without changing default behavior:

  # ~/.mini-agent/config/config.yaml
  log_usage: true

Motivates: tokens/sec throughput stats, per-call cost estimation.

## Downstream use case
minimax-plugin-cc v0.1.3 writes a `timings.ndjson` per-spawn telemetry file for
`/minimax:ask`, `/minimax:review`, `/minimax:adversarial-review`, `/minimax:rescue`.
All three asks above map to fields we currently must emit as `null` / `[]`.
Once they land, we rewire via the already-reserved empty methods in our
`TimingAccumulator` (D2 in our v0.1.3 design).

## What we've tried
- Inspected `agent_run_20260421_214527.log` — `finish_reason` present, no
  served_model, no usage, no per-line timestamps.
- Stdout/stderr parsing — ANSI+emoji+step-indicator human format; unreliable.
- Read-side config.yaml only tells us requested model, not served.

## Priority note
All three are independent. upstream P0 (timestamps) is the smallest and probably
most broadly useful. upstream P1 (served_model) is what mini-agent-plugin-cc
blocks on most. upstream P2 (usage) is strictly opt-in if billing sensitivity
is a concern. Split however suits your triage.
```

**Tracking**: URL recorded in `PROGRESS.md` under new "Upstream tracking" section; `lessons.md §F` tagged `blocked-until-upstream`.

The issue does **not** depend on our code merging. Filing early maximizes upstream feedback window and gives the re-alignment cycle a real status to report.

## 13. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Timing collection in-memory overhead in `callMiniAgent` | Low | Low | all TimingAccumulator ops are O(1) arithmetic on primitive numbers; per-stdout-line overhead bounded by number of stdout lines, not log volume |
| `appendTimingHistory` lock contention blocks caller (v2 clarification) | Low | Low | callers wrap in try/catch; `appendTimingHistory` is synchronous I/O but bounded by 10s lock timeout; called AFTER `callMiniAgent` returns (not in hot path per v2 §5 split); returns `false` on drop so diagnostics can surface it |
| `streamMs` systematically overstates generation time when Mini-Agent auto-retried (new v2) | Medium | Medium | probe P0.2/P0.5 shows 1+2+4s silent retries; padding is invisibly absorbed into `streamMs`. Mitigation: §4 compat callout documents this; cross-plugin aggregates use `totalMs` not `streamMs`; long-term fix depends on upstream P0 (timestamped retries) |
| Concurrent hook runs race on jobs dir prune | Low | Low | `rmSync` is idempotent with `force:true`; ENOENT / EACCES swallowed per D3 |
| `kill(pid, 0)` false positive (pid reused by unrelated process) | Low | Low | old mtime alone does NOT trigger rmSync on non-terminal status; TWO conditions required (ESRCH + mtime>3d). A reused pid belonging to an unrelated live process → `kill(pid,0)` succeeds → we skip → dir preserved. Safe bias: false-retain rather than false-delete |
| ndjson file corrupted mid-write | Low | Medium | crash-recovery prepends `\n` on next append (§6); trim step filters invalid JSON lines before rewriting |
| Gemini re-alignment flags a field-name mismatch we missed | Low | Low | §4 compat callout explicitly lists the 3 fields with semantic drift; reviewers will check the callout first |
| Upstream Mini-Agent declines all three P0/P1/P2 asks | Medium | Medium | `usage` stays `[]`, `tokensPerSec`/`ttftMs`/etc. stay `null` indefinitely; user-facing impact is cosmetic (`—` in aggregate, `fb` column always `—`); no functional loss; `TimingAccumulator` reserved methods still ship as forward-compat scaffolding |
| Synthetic jobIds not in job-control confuse users (C6) | Low | Low | `/minimax:status <synthetic-id>` will return `no such job`; documented in §5 + `/minimax:timing` help text |
| TOCTOU between `readMiniAgentConfig()` and mini-agent's own config read (MiniMax finding) | Low | Low | documented in §5; `requestedModel` recorded at spawn time; user editing config mid-run gets one correct record + one incorrect, neither of which corrupts the ndjson |
