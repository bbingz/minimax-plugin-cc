## 1. Scope (all three ship together in v0.1.3)

1. **Timing telemetry**: `lib/timing.mjs` + `timings.ndjson` history + `/minimax:timing` command. Field names match `gemini-plugin-cc/plugins/gemini/scripts/lib/timing.mjs::TimingAccumulator.build()` output; fields we cannot physically fill (no stream events from Mini-Agent) are explicit `null`.
2. **SessionStart cleanup**: extend `session-lifecycle-hook.mjs` to prune `~/.claude/plugins/minimax/jobs/mj-*/` by mtime > 3 days, AND add SessionEnd per-session-id cleanup mirroring Gemini's pattern adapted to our per-job-directory layout.
3. **Upstream Mini-Agent issue**: file a single issue at `MiniMax-AI/Mini-Agent` with three tiered asks (P0 served_model / P1 usage / P2 per-line timestamps). File **immediately** on v0.1.3 kickoff — not blocked on release.

The three items are mutually independent at the code level but bundled into one release because they share one cross-plugin re-alignment cycle. See §8 for file changes and §9 for test/hard-gate strategy.

## 2. Why these three, why now

- **Timing**: Gemini's P0 finding was "timing 完全缺席". Verified by grep — `timing|appendTiming|TimingAccumulator|timings\.ndjson` has 0 hits anywhere in `plugins/minimax/scripts/`. This is not a stub; it is absent. The user-facing question "is MiniMax actually fast?" is unanswerable without it.
- **SessionStart cleanup**: Gemini's P2 finding flagged `session-lifecycle-hook.mjs` as suspiciously small (30 lines vs their 114). Verified — we do only env injection and session-id file delete. Our `~/.claude/plugins/minimax/jobs/` grows unbounded; Phase 4 never wired cleanup. Left alone, a heavy user hits inode exhaustion eventually.
- **Upstream issue**: P1 (primary-model attestation) is blocked until Mini-Agent exposes a `served_model` field in the log RESPONSE block. Our own v0.1.3 timing cannot fill `usage` / `tokensPerSec` / silent-fallback-detection fields without this upstream change. Filing the issue now maximizes the window for upstream feedback before v0.2.

Gemini's re-alignment specifically compares along these three axes; matching field names and picking 3 days as the mtime threshold closes two of them without a second round-trip. See memory `project-v013-realignment.md`.

## 3. Architectural decisions

### D1. ndjson schema = Gemini's `timing.build()` keys, honest `null` for unfillable fields

**Decision**: our `TimingAccumulator.build()` returns a superset of Gemini's 17-field object. Fields we can compute from post-hoc log parse (no stream events) have real values; fields that require per-token or per-tool stream events are explicit `null`.

**Rationale**:
- Re-alignment requires schema compatibility → field-name drift is the single biggest risk → just name-match.
- Honest `null` communicates "we don't have stream events" clearly; a fabricated value would be more misleading than silence.
- Superset (not subset) means downstream tooling written against Gemini's shape works without branching.

**Trade-off**: our output is visually noisier (many `null` fields). Accepted — readability is fixed at the render layer (`renderHistoryTable` prints `—` for null), not the storage layer.

### D2. TimingAccumulator has all Gemini methods; the stream-path ones are empty no-ops

**Decision**: `onFirstToken`, `onLastToken`, `onToolUseStart`, `onToolResult`, `onRetryStart`, `onRetryEnd`, `onStartupStats`, `onResult`, `dispatchTimingEvent` all exist as empty methods / dead-code paths.

**Rationale**: if Mini-Agent upstream eventually emits stream events (P1/P2 upstream asks), we can wire them in by implementing existing methods rather than refactoring the class. This is the "prepare for upstream success" lever.

**Trade-off**: adds ~40 lines of dead code today. Accepted — significantly less churn than adding the methods later.

### D3. Dual cleanup in session-lifecycle-hook

**Decision**:
- `SessionEnd`: scan `~/.claude/plugins/minimax/jobs/mj-*/`, delete directories where `meta.sessionId === currentSessionId` AND `meta.status ∈ {completed, cancelled, failed}`. Running jobs preserved (Phase 4 detached-worker semantic).
- `SessionStart`: scan same directory, for mtime > 3 days candidates: if `meta.status === "running"`, first `process.kill(meta.pid, 0)` to verify process death; if dead (ESRCH) or already terminal state, `rmSync`. Live process → skip.

**Rationale**:
- SessionEnd handles the happy path (session exits cleanly → its own terminal jobs gone).
- SessionStart handles the sad path (prior session crashed before SessionEnd → its jobs orphaned forever without mtime sweep).
- 3-day threshold matches Gemini's alignment suggestion — zero re-alignment friction.
- `kill(pid, 0)` liveness check preserves the pid-reuse caveat from Phase 4's `cancelJob`: we cannot distinguish a reused pid from the real worker, but `>3 days old` makes reuse-of-the-exact-pid vanishingly unlikely.

**Trade-off**: cleanup runs on every SessionStart, adding a few ms to session boot. Accepted — scan is `listJobs()` + `statSync` per dir, bounded by number of lifetime-recent jobs (typically <100).

### D4. Stale threshold hardcoded at 3 days; not configurable

**Decision**: `const STALE_JOB_THRESHOLD_MS = 3 * 24 * 60 * 60 * 1000` in `session-lifecycle-hook.mjs`. No env override, no config file knob.

**Rationale**: YAGNI. If a user legitimately needs to preserve 5-day-old jobs, they can either (a) touch the directory to bump mtime, (b) edit the constant. Adding an env var is extra surface to document, test, and defend against misuse.

**Trade-off**: one user with a weird workflow might be surprised. Accepted — when that happens, we add the knob.

### D5. `/minimax:timing` offers three views: default history table, `--aggregate` percentiles, `--json` raw

**Decision**: three output modes, no more.

**Rationale**:
- History table answers "what did my recent runs look like?" — the most common question.
- Aggregate answers "what's the p95 latency for `/minimax:ask`?" — the diagnostic/tuning question.
- `--json` lets pipelines / other tools consume raw records.
- Watch mode, SVG export, cross-plugin compare all defer to v0.2 if demand materializes.

### D6. NDJSON file at `${CLAUDE_PLUGIN_DATA}/timings.ndjson`; jobs dir stays at legacy path

**Decision**: new `timings.ndjson` goes to the Claude-Code-standard plugin data path (env `CLAUDE_PLUGIN_DATA` with `~/.claude/plugins/data/minimax-minimax-plugin/` fallback). The existing `~/.claude/plugins/minimax/jobs/` stays where Phase 4 put it.

**Rationale**: Gemini uses the standard path for their ndjson → matching it is cheap compat. Migrating the jobs dir is v0.2 scope (risky, touches every `/minimax:*` command that reads a job), out of v0.1.3 scope.

**Trade-off**: ndjson and jobs live under different roots. Documented in §8 and in `lib/state.mjs` comment.

### D7. Adversarial review emits two ndjson records (one per stance), not one merged

**Decision**: `callMiniAgentAdversarial` runs red then blue sequentially; each spawn gets its own `TimingAccumulator`; each writes its own record with `kind: "adversarial-red"` / `kind: "adversarial-blue"`.

**Rationale**: the two spawns have different prompts, different response times, different failure modes. A merged record would hide that blue is systematically faster than red (or vice versa). Downstream aggregates can filter by `kind` to see per-stance distributions.

**Trade-off**: two rows per adversarial invocation. Accepted — users thinking "one adversarial call should be one row" get the answer "it's two spawns, each costed separately".

## 4. ndjson record shape

```json
{
  "jobId": "mj-<uuid>",
  "kind": "ask" | "review" | "adversarial-red" | "adversarial-blue" | "rescue",
  "ts": "2026-04-23T10:00:00.000Z",
  "timing": {
    "spawnedAt":         "2026-04-23T09:59:55.123Z",
    "firstEventMs":      156,         // spawn -> first stdout line
    "ttftMs":            null,        // no stream events (distinct: firstEvent = first *line*, not first token)
    "streamMs":          12800,       // first stdout line -> last stdout line
    "toolMs":            null,        // no tool events
    "retryMs":           null,        // Mini-Agent internal retries not exposed
    "tailMs":            76,          // last stdout line -> process close
    "totalMs":           13032,       // close - spawnedAt == firstEventMs + streamMs + tailMs (clean exit)
    "promptBytes":       3421,
    "responseBytes":     8912,
    "exitCode":          0,
    "terminationReason": "exit" | "timeout" | "signal" | "error",
    "timedOut":          false,
    "signal":            null,
    "requestedModel":    "MiniMax-M2.7-highspeed",
    "usage":             null,        // [{model,input,output,thoughts}] after upstream P0
    "tokensPerSec":      null,        // derivable from usage once populated
    "coldStartPhases":   null,        // no startup_stats event
    "invariantOk":       null         // cannot verify sum == totalMs without ttft/stream breakdown
  }
}
```

Field semantics match Gemini 1:1 where both sides can compute. For null fields:
- `ttftMs`, `toolMs`, `retryMs`, `coldStartPhases` → no stream events → physically unavailable
- `usage`, `tokensPerSec` → blocked on upstream P0 (served_model + usage counts)
- `invariantOk` → **populable in our case**: `true` if `firstEventMs + streamMs + tailMs === totalMs` on clean exit, `null` otherwise. Simpler invariant than Gemini's (they include ttft/tool/retry in the sum) but still catches clock-skew / accounting bugs.

## 5. TimingAccumulator API

```js
// plugins/minimax/scripts/lib/timing.mjs

export class TimingAccumulator {
  constructor({ spawnedAt = Date.now(), prompt = "" }) { /* init state */ }

  // Active methods (we actually populate these)
  onFirstEvent(t = Date.now())                                 { /* first stdout line — sets firstEvent; also seeds lastEvent */ }
  onStdoutLine(t = Date.now())                                 { /* updates lastEvent on every call */ }
  onClose(t, { exitCode, timedOut, signal })                   { /* process close */ }
  setRequestedModel(name)                                      { /* from config.yaml */ }
  recordResponseBytes(n)                                       { /* from log payload */ }
  build()                                                      { /* returns record.timing; computes invariantOk */ }

  // Empty no-ops (reserved for future upstream stream-event wiring — D2)
  onFirstToken()     { /* reserved */ }
  onLastToken()      { /* reserved */ }
  onToolUseStart()   { /* reserved */ }
  onToolResult()     { /* reserved */ }
  onRetryStart()     { /* reserved */ }
  onRetryEnd()       { /* reserved */ }
  onStartupStats()   { /* reserved */ }
  onResult()         { /* reserved */ }
}

export function dispatchTimingEvent(event, timing) { /* no-op stub, reserved */ }

// Aggregation + render helpers (port of Gemini's equivalents, adapted for our nulls)
export function percentile(values, p);
export function computeAggregateStats(records);
export function filterHistory(records, { kind, last, since });
export function renderHistoryTable(rows);
export function renderAggregateTable(stats, { kind });
export function renderStatusSummaryLine(timing);   // nice-to-have for rescue/status output
export function formatMs(ms);
```

### Integration in `callMiniAgent` (pseudocode)

```js
export async function callMiniAgent({ prompt, timeout, jobId, kind, onStdoutLine, ...rest }) {
  const timing = new TimingAccumulator({ spawnedAt: Date.now(), prompt });
  timing.setRequestedModel(await readRequestedModel()); // from ~/.mini-agent/config/config.yaml

  let firstEventSeen = false;
  const wrappedOnStdoutLine = (line) => {
    if (!firstEventSeen) { timing.onFirstEvent(); firstEventSeen = true; }
    timing.onStdoutLine();  // always update lastEvent for streamMs/tailMs calc
    onStdoutLine?.(line);
  };

  const { exitCode, timedOut, signal } = await spawnWithHardTimeout({
    /* ... */,
    onStdoutLine: wrappedOnStdoutLine,
  });
  timing.onClose(Date.now(), { exitCode, timedOut, signal });

  // Log parse happens after close; its cost is NOT in tailMs (tail = close - lastEvent).
  // Log-parse latency could be tracked in v0.2 if it becomes interesting.
  const parsed = await parseLogFileAndClassify(/* ... */);
  timing.recordResponseBytes(parsed.responseBytes || 0);

  // Fire-and-forget record append (D1: honest null for unfillable)
  try { appendTimingHistory({ jobId, kind, ts: new Date().toISOString(), timing: timing.build() }); }
  catch (e) { /* never let telemetry block the primary result */ }

  return parsed;
}
```

## 6. appendTimingHistory (`lib/state.mjs`)

Port of Gemini's `appendTimingHistory` (their `lib/state.mjs` lines 274+):

- Path: `${CLAUDE_PLUGIN_DATA}/timings.ndjson` or fallback `~/.claude/plugins/data/minimax-minimax-plugin/timings.ndjson`.
- Lock: O_EXCL `timings.ndjson.lock` sibling, 10s acquire timeout, 30s stale-lock reclaim.
- Crash recovery: if file does not end with `\n`, prepend one before appending the new record.
- Retention: if file size > `TIMING_MAX_BYTES = 10 * 1024 * 1024`, keep only the newer half of valid JSON lines, atomic rename via `.tmp`.
- Lock acquire timeout → log to stderr and drop the record (never throw — telemetry must not affect primary result).

## 9. Test strategy + T14 hard gate

### Unit tests

**`timing.test.mjs`** (new, ~10 tests)
- `TimingAccumulator` spawn → firstEvent → multiple onStdoutLine → close yields correct `build()` values
- `firstEventMs`, `streamMs`, `tailMs` all `null` when spawn → close with no stdout
- `streamMs` = 0 when only one stdout line seen (first == last)
- `tailMs` clamped ≥ 0 when clock returns to earlier time mid-call
- `invariantOk` true when `firstEventMs + streamMs + tailMs === totalMs` on clean exit
- `invariantOk` null when terminationReason !== "exit"
- `onClose` covers exit / timeout / signal terminationReason values
- `recordResponseBytes` accumulates across calls
- Empty no-op methods (`onFirstToken` etc.) are callable and do not throw
- `computeAggregateStats` n<20 → p95 null; n<100 → p99 null
- `computeAggregateStats` fallback rate = 0 when all records have `usage: null`
- `renderHistoryTable` prints `—` for null fields
- `filterHistory` kind/since/last filters combine correctly

**`state.test.mjs`** (extend, +5 tests)
- `appendTimingHistory` basic append → each line JSON-parseable
- 10MB cap triggers half-trim → file size < cap → retained lines are newer half
- Crash recovery: file without trailing `\n` → next append prepends one
- Concurrent O_EXCL lock: two appenders → both eventually succeed or second returns false after 10s
- Lock path resolves to `CLAUDE_PLUGIN_DATA` when env is set, fallback otherwise

**`session-lifecycle-hook.test.mjs`** (new, ~6 tests)
- SessionStart: mtime > 3d + status terminal → rmSync invoked
- SessionStart: mtime > 3d + status running + `kill(pid,0)` throws ESRCH → rmSync invoked
- SessionStart: mtime > 3d + status running + kill(pid,0) succeeds (alive) → skip
- SessionStart: mtime < 3d → preserved regardless of status
- SessionEnd: session-matched terminal job → rmSync; session-matched running job → preserved
- rmSync throws EACCES → hook exits 0, does not propagate

## 13. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Timing collection slows `callMiniAgent` | Low | Low | all timing ops are O(1) arithmetic; no synchronous I/O in hot path; ndjson append is fire-and-forget wrapped in try/catch |
| Concurrent hook runs race on jobs dir prune | Low | Low | `rmSync` is idempotent; ENOENT / EACCES swallowed per D3 |
| `kill(pid, 0)` false positive (pid reused) | Very low | Low | only triggered on mtime > 3d; pid reuse within exact match at 3-day scale is vanishingly rare; fallback behavior is "skip" which is safe |
| ndjson file corrupted mid-write | Low | Medium | D6 crash-recovery rewrite prepends `\n`; parser filters invalid JSON lines during trim |
| Gemini re-alignment flags a field-name mismatch we missed | Medium | Low | explicit `refs/gemini-plugin-cc:plugins/gemini/scripts/lib/timing.mjs@d96517` cross-reference in our `TimingAccumulator` JSDoc so reviewers can diff |
| Upstream Mini-Agent declines P0 | Medium | Medium | `usage` / `tokensPerSec` stay null indefinitely; user-facing impact is cosmetic (`—` in aggregate); no functional loss |
